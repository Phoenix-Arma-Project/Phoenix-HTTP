## 📖 О проекте

**Phoenix HTTP** — это современный HTTP-клиент для выделенных серверов (dedicated servers) Arma 3, созданный как нативное расширение на **C# Native AOT**. Он позволяет вашим миссиям и фреймворкам взаимодействовать с REST API и веб-сервисами, совершенно не загружая и не задерживая игровой цикл.

Он состоит из двух частей, работающих совместно:

* **нативное расширение** (`PhoenixHttp_x64.dll`), выполняющее фактическую работу по HTTP-запросам асинхронно в фоновом пуле потоков;
* **тонкая SQF-обертка** (`PHTTP_fnc_Request`), которая формирует запросы, отправляет их через `callExtension` и возвращает ответы в ваш код через функцию обратного вызова (callback).

Поскольку расширение никогда не блокирует игровой движок, а ответ парсится в HashMap, значения из которого можно читать с помощью простых точечных ключей, работа с веб-API из SQF перестает быть мучительной.

## ✨ Возможности

* ⚡ **Плоские JSON-ответы** — в Arma 3 нет встроенной поддержки JSON, поэтому Phoenix HTTP парсит ответ и возвращает HashMap, из которого можно читать данные с помощью точечных ключей. Например, тело ответа сервера `{"user":{"id":123}}` будет доступно как `_response get "body.user.id"`. Сторонний SQF JSON-парсер не требуется.
* 🔁 **Массивы остаются итерируемыми** — объекты преобразуются в плоскую структуру с точечными ключами, но **массивы остаются настоящими массивами SQF**, что позволяет обходить их через `forEach`. Каждым объектом внутри массива является готовый к использованию HashMap.
* 🗃️ **Безопасная обработка null** — значения `null` из JSON исключаются из результатов, поэтому отсутствующее поле просто не создается, а функция `getOrDefault` всегда работает корректно. Ваши скрипты больше не упадут из-за неожиданных значений null.
* ⏳ **Полностью асинхронный и неблокирующий режим** — запросы выполняются в фоновом пуле потоков и возвращаются через обратный вызов движка (engine callback). Сервер никогда не зависает и не теряет TPS в ожидании ответа от API.
* 🧵 **Ограничение конкурентности** — настраиваемая очередь ограничивает количество одновременно выполняемых запросов, предотвращая истощение ресурсов сервера при резком всплеске вызовов.
* 🧩 **Макро-переменные окружения** — плейсхолдеры вида `{{NAME}}` заменяются значениями из `config.json`, благодаря чему ключи API и базовые URL-адреса хранятся в конфигурации, а не жестко прописываются в файлах миссии.
* 📦 **Автоматическое разбиение на чанки** — `callExtension` имеет строгое ограничение на размер ответа. Phoenix HTTP автоматически разбивает большие ответы на фрагменты (чанки) и собирает их обратно на стороне SQF.
* 🚀 **Ноль внешних зависимостей времени выполнения** — расширение скомпилировано с помощью Native AOT, поэтому на сервере **не требуется установленная среда выполнения .NET**. Просто добавьте мод и пользуйтесь.
* 📝 **Логирование в файлы** — вся сетевая активность, разбиение на чанки и ошибки записываются в лог-файлы (по одному на каждый запуск) для простой отладки.

## 📋 Требования

| Требование | Примечания |
| :--- | :--- |
| **Выделенный сервер на Windows** | Расширение собрано исключительно под **win-x64** (`PhoenixHttp_x64.dll`). Серверы на Linux **не** поддерживаются. |
| **Сервер Arma 3 ≥ 2.02** | Требуется для использования `callExtension` с передачей параметров в виде массива. |
| Среда выполнения .NET | **Не требуется** — расширение полностью автономно (Native AOT). |

## 🛠️ Установка

1. Скачайте последнюю сборку со страницы [Releases](../../releases).
2. Распакуйте папку `@PhoenixHTTP` в корневую директорию вашего сервера Arma 3.
3. Добавьте `-serverMod=@PhoenixHTTP` в параметры запуска вашего сервера.

Структура папки `@PhoenixHTTP` выглядит следующим образом:

```
@PhoenixHTTP/
├── addons/
│   └── phoenixhttp_main.pbo   # SQF-обертка (PHTTP_fnc_Request, обработчик коллбэков)
├── PhoenixHttp_x64.dll        # нативное расширение
├── config.json                # ваша конфигурация (скопированная из config.example.json)
└── logs/                      # создается во время работы, по одному файлу на каждый запуск сервера
```

Файлы `config.json` и папка `logs/` располагаются рядом с DLL.

## ⚙️ Настройка (`config.json`)

Все параметры являются необязательными; если файл или конкретный ключ отсутствует, используются значения по умолчанию. Числовые значения имеют нижнее ограничение в `1`, поэтому некорректное значение не сломает работу расширения.

```json
{
    "maxConcurrentRequests": 8,
    "requestTimeoutSeconds": 30,
    "chunkSize": 8192,
    "debug": false,
    "environments": {
        "API_BASE": "https://api.yoursite.com/v1",
        "API_KEY": "secret_token_123"
    }
}
```

| Ключ | Тип | По умолчанию | Описание |
| :--- | :--- | :--- | :--- |
| `maxConcurrentRequests` | Целое число | `8` | Максимальное количество одновременно выполняемых запросов. Дополнительные запросы будут ожидать в очереди. |
| `requestTimeoutSeconds` | Целое число | `30` | Таймаут одного запроса. Запрос, превысивший это время, завершится с ошибкой `Timeout`. |
| `chunkSize` | Целое число | `8192` | Максимальный размер (в байтах) каждого фрагмента (чанка) ответа, передаваемого в SQF. |
| `debug` | Логический | `false` | При значении `true` подробные логи уровня `DEBUG` записываются в лог-файл. |
| `environments` | Объект | `{}` | Именованные переменные, подставляемые в запросы с помощью синтаксиса макросов `{{NAME}}`. |

Конфигурацию можно перезагрузить прямо во время работы сервера без его перезапуска — см. раздел **Для опытных пользователей: сырой протокол `callExtension`** ниже.

## 📚 Справочник API (API Reference)

Обертка предоставляет единственную функцию — `PHTTP_fnc_Request`. Она принимает один HashMap с описанием запроса и немедленно возвращает идентификатор транзакции (строку). Ответ будет доставлен позже в ваш `callback`.

```sqf
private _id = [_request] call PHTTP_fnc_Request;
```

### 📤 Параметры запроса (Request keys)

| Ключ | Тип | По умолчанию | Описание |
| :--- | :--- | :--- | :--- |
| `method` | Строка | `"GET"` | HTTP-метод. Допускается любой метод (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD` и др.); регистр автоматически приводится к верхнему. |
| `url` | Строка | `""` | Целевой URL. Поддерживает макросы `{{macros}}`. |
| `parser` | Строка | `"json"` | `"json"` парсит и делает плоским тело ответа; **любое другое значение** (например, `"raw"`) возвращает тело ответа в виде обычной строки. |
| `headers` | HashMap | `createHashMap` | Заголовки запроса в формате «имя → значение». Значения поддерживают макросы `{{macros}}`. |
| `query` | HashMap | `createHashMap` | Параметры строки запроса в формате «имя → значение». Они кодируются (URL-encode) и добавляются к URL. Значения поддерживают макросы `{{macros}}`. |
| `body` | Строка или HashMap | `""` | Полезная нагрузка запроса. **HashMap** сериализуется в JSON и отправляется с заголовком `Content-Type: application/json`. **Строка** отправляется как есть — установите `Content-Type` самостоятельно в `headers`. Поддерживает макросы `{{macros}}`. |
| `callback` | Код | `{}` | Выполняется, когда ответ готов, получая HashMap с ответом в качестве `_this select 0`. **Опустите этот параметр для режима «отправил и забыл» (fire-and-forget)**, при котором ответ отбрасывается и не запрашивается из расширения. |

### 📥 Параметры ответа (Response keys)

Функция обратного вызова (callback) принимает один HashMap. Его ключи являются **плоскими**, поэтому вы можете читать вложенные значения через точечные ключи, избегая цепочек вызовов `get`.

| Ключ | Тип | Описание |
| :--- | :--- | :--- |
| `success` | Логический | `true`, если от сервера был получен ответ (с **любым** HTTP-статусом, включая `404` или `500`). `false` возвращается только при транспортных сбоях (сетевая ошибка, таймаут, некорректный запрос). |
| `status_code` | Число | При успешном выполнении — HTTP-код состояния (`200`, `404` и т.д.). При сбое — **отрицательный код ошибки** (см. ниже). |
| `headers.<имя>` | Строка | Заголовки ответа, приведенные к плоскому виду с именами в нижнем регистре (например, `_response get "headers.content-type"`). Это **не** вложенный HashMap. |
| `body` / `body.<путь>` | Любой | Тело ответа (см. раздел **Тело ответа: как работает приведение к плоскому виду** ниже). Отсутствует, если запрос полностью завершился сбоем. |

### ⚠️ Коды ошибок (`status_code` при значении `success` равном `false`)

| Код | Значение |
| :--- | :--- |
| `-1` | Сеть недоступна (ошибка DNS, соединение отклонено, хост недоступен). |
| `-2` | Превышено время ожидания запроса (истек `requestTimeoutSeconds`). |
| `-4` | Ошибка сериализации/парсинга ответа (например, невалидный JSON, когда параметр `parser` равен `"json"`). |
| `-100` | Неизвестная ошибка (например, некорректный URL или метод). Подробности смотрите в лог-файле. |

### 🧬 Тело ответа: как работает приведение к плоскому виду (flattening)

Тело ответа нормализуется для SQF по одному простому правилу:

* **Объекты** превращаются в **плоские точечные ключи**. `{"user": {"name": "Bob"}}` превращается в `body.user.name`, значение которого можно прочитать одним вызовом `get`.
* **Массивы** сохраняются как настоящие **массивы SQF**, чтобы они оставались итерируемыми с помощью `forEach`. Каждый объект *внутри* массива сам по себе является плоским HashMap, доступ к которому осуществляется сначала по индексу, а затем по ключу.
* Значения **`null`** опускаются — ключ просто не будет присутствовать в ответе.

```sqf
// Ответ сервера: { "users": [ {"name": "Bob"}, {"name": "Alice"} ], "total": 2 }

_response get "body.total";                       // 2

private _users = _response get "body.users";      // настоящий массив SQF
{
    diag_log (_x get "name");                     // каждый элемент является плоским HashMap
} forEach _users;
```

Если параметр `parser` отличается от `"json"`, ключ `body` возвращается как обычная строка.

## 💻 Примеры использования

### 📍 GET-запрос с параметрами запроса

```sqf
private _request = createHashMapFromArray [
    ["method", "GET"],
    ["url", "{{API_BASE}}/users/1"],
    ["query", createHashMapFromArray [
        ["include_stats", "true"]
    ]],
    ["callback", {
        params ["_response"];

        if !(_response get "success") exitWith {
            diag_log "[MyMod] Request failed (network/timeout).";
        };

        if ((_response get "status_code") != 200) exitWith {
            diag_log format ["[MyMod] API error: %1", _response get "status_code"];
        };

        // Доступ к вложенным полям через точечные ключи.
        // Использование getOrDefault безопасно, так как значения null опускаются.
        private _name = _response getOrDefault ["body.user.name", "Unknown"];
        diag_log format ["[MyMod] User: %1", _name];
    }]
];

[_request] call PHTTP_fnc_Request;
```

### 📍 POST-запрос с телом в формате JSON

```sqf
private _request = createHashMapFromArray [
    ["method", "POST"],
    ["url", "{{API_BASE}}/users"],
    ["headers", createHashMapFromArray [
        ["Authorization", "Bearer {{API_KEY}}"]
        // Content-Type устанавливается автоматически, так как параметр 'body' является HashMap.
    ]],
    ["body", createHashMapFromArray [
        ["name", "John Doe"],
        ["role", "admin"],
        ["tags", ["pvp", "staff"]]   // вложенные массивы и карты тоже сериализуются
    ]],
    ["callback", {
        params ["_response"];
        diag_log format ["[MyMod] Created -> %1", _response get "status_code"];
    }]
];

[_request] call PHTTP_fnc_Request;
```

### 📍 Сырое тело запроса и пользовательский Content-Type

```sqf
private _request = createHashMapFromArray [
    ["method", "POST"],
    ["url", "{{API_BASE}}/webhook"],
    ["parser", "raw"],                                  // получить ответ как обычную строку
    ["headers", createHashMapFromArray [
        ["content-type", "text/xml"]                    // устанавливайте вручную при передаче тела строкой
    ]],
    ["body", "<event>player_joined</event>"],
    ["callback", {
        params ["_response"];
        diag_log (_response getOrDefault ["body", ""]);
    }]
];

[_request] call PHTTP_fnc_Request;
```

### 📍 Режим «отправил и забыл» (Fire-and-forget)

Если опустить `callback`, ответ никогда не будет запрашиваться из расширения. Это полезно для телеметрии или вебхуков, когда результат выполнения вам не важен.

```sqf
[createHashMapFromArray [
    ["method", "POST"],
    ["url", "{{API_BASE}}/heartbeat"]
]] call PHTTP_fnc_Request;
```

## 🧩 Макросы

Любой токен вида `{{NAME}}` заменяется соответствующим значением из раздела `environments` в файле `config.json`. Макросы раскрываются в **URL**, **значениях заголовков**, **параметрах запроса** и **теле запроса**. Неизвестный макрос оставляется без изменений (включая фигурные скобки), благодаря чему опечатку будет легко заметить.

```json
"environments": { "API_BASE": "https://api.example.com", "API_KEY": "abc123" }
```
```sqf
["url", "{{API_BASE}}/status"]                       // -> https://api.example.com/status
["headers", createHashMapFromArray [["Authorization", "Bearer {{API_KEY}}"]]]
```

## ⚙️ Для опытных пользователей: сырой протокол `callExtension`

Функция `PHTTP_fnc_Request` построена поверх этого набора низкоуровневых команд. Вы можете вызывать их напрямую при необходимости. Каждый ответ представляет собой строку вида `status:data`, где `status` — это `success` или `error`. Разбиение строки следует выполнять **только по первому двоеточию**, поскольку данные (`data`) сами могут содержать двоеточия.

| Команда | Аргументы | Возвращает | Описание |
| :--- | :--- | :--- | :--- |
| `request:create` | `[method, url]` | `success:<id>` | Создает транзакцию и возвращает её ID. |
| `request:header` | `[id, key, value]` | `success:` | Добавляет заголовок запроса. |
| `request:query` | `[id, key, value]` | `success:` | Добавляет параметр строки запроса. |
| `request:body` | `[id, chunk]` | `success:` | Добавляет фрагмент (чанк) к телу запроса. |
| `request:send` | `[id, parser, needsResponse]` | `success:<id>` | Отправляет запрос. Параметр `parser` может принимать значения `"json"`/`"raw"`; параметр `needsResponse` — `"true"`/`"false"`. |
| `request:get` | `[id, chunkIndex]` | `success:<chunk>` | Возвращает один чанк готового ответа. |
| `request:delete` | `[id]` | `success:` | Удаляет транзакцию. Вызывайте после прочтения всего ответа. |
| `request:clear` | `[]` | `success:` | Удаляет все транзакции. |
| `config:reload` | `[]` | `success:` | Перечитывает `config.json` прямо во время работы (ограничение конкурентности, таймауты, макросы, режим отладки). |

Когда `needsResponse` установлен в `"true"`, расширение генерирует игровое событие `ExtensionCallback` с аргументами `["PhoenixHttp", "response", "[""<id>"", <chunkCount>]"]`, как только ответ будет полностью подготовлен. После этого вам необходимо запросить все чанки через `request:get`, склеить их и применить `call compile` к полученному результату для воссоздания HashMap ответа (именно так работает поставляемая в комплекте обертка).

```sqf
private _reply = ("PhoenixHttp" callExtension ["request:create", ["GET", "https://api.ipify.org?format=json"]]) select 0;
private _sep = _reply find ":";
private _status = _reply select [0, _sep];
private _id = _reply select [_sep + 1];     // надежный способ: деление только по первому двоеточию

if (_status isEqualTo "success") then {
    "PhoenixHttp" callExtension ["request:send", [_id, "json", "true"]];
};
```

Для перезагрузки конфигурации «на лету»:

```sqf
"PhoenixHttp" callExtension ["config:reload", []];
```

## 🔧 Сборка из исходников

Вам понадобятся **.NET 10 SDK** и инструмент **[HEMTT](https://github.com/BrettMayson/HEMTT)**.

```sh
# Сборка аддона (PBO)
hemtt build

# Сборка нативного расширения (Windows, Native AOT)
dotnet publish extension/PhoenixHTTP/PhoenixHTTP.csproj -c Release

# Запуск юнит-тестов для логики сериализации и разбиения на чанки
dotnet test extension/PhoenixHTTP.Test/PhoenixHTTP.Test.csproj
```

Публикация расширения в режиме AOT (Native AOT) требует наличия компилятора MSVC в среде Windows. Система непрерывной интеграции (CI) собирает аддон под Linux, а расширение — под Windows, после чего упаковывает готовый мод `@PhoenixHTTP`.
